# /workflows-plan

将功能描述、bug报告或改进想法转换为结构化的 markdown 计划文档。

## 流程

1. **仓库研究** - 理解项目约定和模式
2. **计划结构** - 起草清晰、可执行的计划
3. **选择详细程度** - **必须询问用户**选择详细程度，等待回复后再继续
4. **创建计划文件** - 写入 `plans/<issue_title>.md`

## 执行规则

**在创建计划文件之前，必须先询问用户选择详细程度：**

请选择计划的详细程度：
- **MINIMAL** - 快速计划（问题描述+验收标准）
- **MORE** - 标准计划（+技术考虑+风险）
- **A LOT** - 详尽计划（+实施阶段+替代方案）

**等待用户回复后，再根据选择的级别创建计划文档。**

## 详细程度

### MINIMAL
- 问题描述
- 基本验收标准
- 必要上下文

### MORE
- 详细背景和动机
- 技术考虑
- 成功指标
- 依赖和风险

### A LOT
- 详细实施阶段
- 考虑的替代方案
- 资源需求和时间线
- 风险缓解策略

## 输出

计划文件保存到 `plans/<issue_title>.md`

## 后续步骤

1. 运行 `/workflows-work` 执行计划
2. 实施后运行 `/workflows-review`
3. 运行 `/workflows-compound` 记录学习
